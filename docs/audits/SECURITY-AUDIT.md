# Audit Sécurité — Module d'enrichissement OSINT

**Auditeur** : Security Guardian
**Date** : 2026-05-11
**Périmètre** : Module d'enrichissement OSINT (~40 fichiers backend + frontend)
**Méthodologie** : revue de code statique, lecture des diffs vs `main`, vérification des 6 ADR, analyse du flux de données externe → DB → DOM, vérification de l'auth/CORS/headers.

## Synthèse exécutive

État global : **ORANGE — déploiement BLOQUÉ tant que les deux failles CRITIQUES ne sont pas corrigées.** Le module est globalement bien conçu (Prisma protège des injections SQL, validation zod sur les routes POST, audit RGPD avec expurgation de données personnelles, garde-fou ADR-006 effectif). Toutefois, deux vulnérabilités critiques exposent le projet à (1) une **injection SPARQL** côté Wikidata permettant de modifier la requête envoyée à un serveur tiers et (2) une **violation du principe RGPD de traçabilité des traitements** via le `onDelete: Cascade` sur l'audit. Quatre vulnérabilités IMPORTANTES (XSS via href externe non assaini, fuite d'email dans le User-Agent, route graphe entièrement publique, CORS par défaut localhost en prod) doivent être traitées avant le merge. Plusieurs points mineurs concernent l'hygiène du code et la documentation légale manquante.

## RGPD / Conformité

### Conforme

- **Expurgation des données personnelles dans l'audit** (`audit.js:19-67`) : regex email + téléphone FR, IP tronquée IPv4/IPv6. ✓
- **Garde-fou `qualiteInfluencePublique` côté serveur** : `import-enrichissement.js:169-173` lève une `Error` avant toute écriture DB. ✓ ADR-006 effective.
- **Affichage public du badge de qualité** : implémenté dans le frontend (transparence éditoriale). ✓
- **Cadastre nominatif explicitement hors scope** : `ign-carto-cadastre.js:23-25` ne tente pas d'accéder aux fichiers MAJIC. ✓ ADR-003 respectée.
- **DVF sans noms de propriétaires** : `ign-dvf.js:17-19` documenté, respecté par le mapping. ✓
- **`REDACTED FOR PRIVACY` filtré côté RDAP** : `rdap.js:60-68, 225` n'expose pas les titulaires anonymisés. ✓

### Non-conforme ou à risque

- **SEC-C-02 : audit non immuable** — viole art. 5.1.f (intégrité) et l'obligation d'accountability art. 5.2 RGPD. Critique.
- **SEC-I-03 : exposition de données personnelles sur le graphe public** — viole le principe de minimisation art. 5.1.c quand des entités `EN_ATTENTE` non encore validées sont accessibles à un anonyme.
- **SEC-M-03 : mentions légales et politique de protection des données absentes** — obligatoire (art. 13 RGPD, information des personnes concernées). Non-conformité directe avant toute mise en ligne nominative.
- **SEC-M-04 : pas de procédure d'exercice du droit d'opposition** — art. 21 RGPD reste applicable pour les personnes publiques traitées sous art. 85.
- **SEC-M-07 : référence légale art. 80 LIL douteuse** — à faire valider par un juriste avant publication.

### Bases légales à documenter explicitement

L'art. 85 RGPD permet aux États membres de prévoir des exceptions pour le journalisme et la recherche. **En droit français, cette exception est conditionnée** — elle ne supprime pas l'art. 13 (information), l'art. 21 (opposition), ni l'art. 15 (accès). Le projet doit publier :

1. Une page d'information conforme art. 13 RGPD (qui, pourquoi, durée, droits, contact DPO).
2. Une procédure d'exercice des droits.
3. Une analyse d'impact (AIPD) si traitement massif — peut être différée tant que le nombre d'entités < 1000, mais à anticiper.

## Points forts du code

- **Architecture connecteurs propre** : `BaseConnecteur` avec cache + rate-limit + timeout + User-Agent — pattern correct et maintenable.
- **Token bucket sans `setInterval`** : `rate-limit.js` utilise `process.hrtime.bigint()` — élégant, testable, ne fuit pas de timer.
- **Dégradation gracieuse du cache** : bascule fichier → mémoire si disque inaccessible (`cache.js:35-51`) — bonne résilience Infomaniak mutualisé.
- **Sources marquées par champ** : `marquerProvenance` impose la traçabilité de chaque donnée — design RGPD-friendly.
- **Validation zod systématique sur les routes POST** : présent et bien structuré (sauf SEC-I-05).
- **Prisma `$transaction` pour l'import** : atomicité respectée, pas de fragmentation possible.
- **Tronquage IP IPv4 + IPv6** : implémentation correcte dans `audit.js:35-53`.
