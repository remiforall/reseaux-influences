# Audit Accessibilité — Module d'enrichissement OSINT

**Auditeur** : Accessibility Champion
**Date** : 2026-05-11
**Cible** : WCAG 2.1 AAA + RGAA 4.1.2
**Périmètre** : 17 fichiers UI (pages `Enrichissement`, `Graphe`, 12 composants enrichissement/graphe, Navbar, App, index.css)
**Outils** : revue de code statique uniquement (aucun test navigateur ni screen reader réel). Les calculs de contraste utilisent les valeurs Tailwind v3 (palette 2024).

---

## Synthèse exécutive

Le module présente une **base d'accessibilité solide** : sémantique HTML correcte (`<fieldset>`, `<legend>`, `<label htmlFor>`, `<dl>`, `<table>` + `<caption>`), labels associés systématiquement, gestion explicite du `prefers-reduced-motion` dans `index.css` et dans `GrapheD3` (alphaDecay 0.9), palette CUD-safe sur tous les badges (triple redondance couleur + icône + texte), et alternative tableau du graphe.

**Conformité AAA estimée : 60–65 %.** Les barrières AAA sont concentrées sur (1) les contrastes de texte d'agrément (`text-gray-500`, `text-secondary`, `text-amber-800`, `hover:text-accent` de la Navbar) qui ne tiennent pas le 7:1, (2) l'absence totale de skip link, (3) le `role="dialog"` du `FormulaireImport` sans focus trap ni gestion `Escape`, (4) les cibles tactiles à 36 px dans `FiltresGraphe`, (5) un anti-pattern React `document.getElementById` dans `GrapheD3` pour le bouton « Recentrer ». Aucune barrière n'empêche totalement l'usage du module, mais plusieurs créent un inconfort majeur pour utilisateurs malvoyants, dys, et navigation clavier exclusive.

## Recommandations prioritaires

Par ordre d'impact utilisateur et de coût d'implémentation :

1. **Skip link « Aller au contenu »** (I-01) — 15 min de code, impact majeur pour tous les utilisateurs clavier. Ajouter dans `Navbar.jsx` ou en haut de `App.jsx`, avec `id="contenu-principal"` sur `<main>`.
2. **Focus trap + Escape dans `FormulaireImport`** (B-01) — 2 h de code, impact majeur clavier + screen reader. Extraire un composant `<Dialog>` réutilisable (sera réutilisé partout dans le projet).
3. **Remplacer le coupling `getElementById('btn-recentrer-graphe')`** (B-02) — 30 min. `forwardRef` + `useImperativeHandle` propre.
4. **Augmenter les contrastes texte sur fond clair** (I-04, A-01, A-02, A-03) — 1 h. Modifier `tailwind.config.js` pour assombrir `secondary`, remplacer toutes les occurrences `text-gray-500` → `text-gray-600`, `text-amber-800` → `text-amber-900`, `text-red-700` → `text-red-800`.
5. **Cibles tactiles 44 px partout** (I-06) — 15 min. Remplacer `min-h-[36px]` → `min-h-[44px]` dans `FiltresGraphe`.
6. **Redirection de focus sur erreur de formulaire** (I-10) — 30 min. `useRef` sur l'input + `.focus()` dans le handler de validation.
7. **Désactivation du `tabIndex` redondant sur `<tr>`** (B-03) — 5 min. Supprimer `tabIndex={0}` et `onKeyDown` de la `<tr>`, laisser les boutons gérer.
8. **Conditionner toutes les animations D3 par `prefers-reduced-motion`** (A-07, A-08) — 30 min. Helper `getTransitionDuration()` partagé.
9. **Centraliser les `aria-live`** (A-13) — 1 h. Toutes les annonces dynamiques passent par `RegionAnnonces`.
10. **Glossaire et `<abbr>`** (A-21) — 2 h. Page `/glossaire` + remplacement systématique des sigles dans le texte courant.

**Effort total estimé pour atteindre AAA stricte sur le module enrichissement** : 12-16 h de dev. Effort pour atteindre AA stricte (déjà excellent) : 4-6 h.

## Points forts à conserver

- **Palette CUD-safe disciplinée** : tous les badges (BadgeProvenance, StatutConnecteur, TypeBadge, StatutBadge) suivent le pattern triple redondance couleur + icône + texte. Excellent travail, à industrialiser dans un composant partagé.
- **Sémantique HTML rigoureuse** : `<fieldset>`/`<legend>` pour les groupes de checkboxes, `<dl>` pour les paires clé/valeur, `<caption>` + `<th scope>` pour le tableau, `<time datetime>` pour les dates.
- **Labels explicites** : tous les champs de formulaire ont un `<label htmlFor>` lié, avec `aria-describedby` vers les messages d'aide et d'erreur. Astérisque visuel rouge `*` + texte `sr-only` « (champ obligatoire) ».
- **`prefers-reduced-motion` géré globalement** dans `index.css` + spécifiquement dans `GrapheD3` (alphaDecay 0.9 → convergence immédiate).
- **Alternative tableau du graphe** (`TableauGraphe`) — réponse exemplaire au critère 1.1.1 sur la visualisation. Tri clavier OK.
- **Note RGPD/base légale en `<aside role="note">`** placée AVANT le formulaire — pratique RGAA exemplaire.
- **Confirmation explicite avant import** (checkbox + bouton désactivé tant que non cochée) — répond au critère 3.3.4 AA et au-delà.
- **Combobox `SelecteurEntite`** : pattern ARIA combobox correctement implémenté (`role="combobox"`, `aria-autocomplete="list"`, `aria-expanded`, `aria-activedescendant`, `role="listbox"`, `role="option"` `aria-selected`). Navigation flèches OK.
- **`<html lang="fr">`** + `<meta name="viewport">` sans `user-scalable=no` — base saine.
- **Focus restauration** sur le toggle après changement de vue graphe/tableau (`toggleRef.current?.focus()`), focus sur le premier résultat après recherche.
- **`aria-pressed`** correctement utilisé sur le toggle de vue, `aria-expanded`/`aria-controls` sur le bouton « Voir les détails ».
- **`aria-busy`** sur le bouton de soumission pendant la recherche.

**Verdict global** : le module est **conçu avec une vraie sensibilité accessibilité**. Les barrières AAA restantes sont essentiellement cosmétiques (contrastes à ajuster, cibles tactiles à élargir) ou liées à un composant Dialog à factoriser. Avec un sprint d'environ 2 jours-dev, le module passerait à 90 %+ de conformité AAA, et le composant `<Dialog>` créé pourrait servir le reste du projet.
